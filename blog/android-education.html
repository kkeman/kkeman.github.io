개발 조직에서 사내 기술 교육을 설계하는 법

비전공자부터 다른 플랫폼 개발자까지, 다양한 배경의 수강생에게 기술을 가르칠 때 효과적인 커리큘럼 설계 원칙들.

교육의 목표부터 정하라

"Android 개발자를 만드는 것"과 "Android 환경을 이해하는 사람을 늘리는 것"은 완전히 다른 목표입니다. 목표에 따라 커리큘럼 깊이가 달라집니다.

SDK 팀 밖의 사람들 — 백엔드 개발자, QA, 기획자 — 이 Android의 기본 구조를 이해하면 커뮤니케이션 비용이 줄어듭니다. 크래시 리포트를 보고 "이건 RecyclerView Adapter 문제 같다"고 말할 수 있는 QA와, "앱이 죽어요"만 말하는 QA는 차원이 다릅니다.

수강생 분석이 먼저

같은 교육이라도 수강생 배경에 따라 접근이 달라야 합니다:

  • Java/Spring 백엔드 개발자 — Kotlin 문법은 빠르게 넘어가고 Android 특수성(Activity 생명주기, UI 스레드)에 시간을 투자
  • 프론트엔드 개발자 — UI 구성 개념은 익숙하니 Layout XML과 React의 차이점 중심
  • 비개발자(QA, 기획) — 코드보다 "왜 이렇게 동작하는지" 개념 설명에 집중

8주 커리큘럼 설계 예시

1~2주: 환경 세팅 + 첫 빌드

Android Studio 설치, 에뮬레이터 세팅, Hello World. 자기 폰에서 앱이 실행되는 경험이 동기를 만듭니다. 첫 수업 전에 환경 세팅을 완료하게 하세요 — 세팅에 수업 시간을 쓰면 2시간이 날아갑니다.

3~4주: UI + 리스트

Layout XML, RecyclerView. 실습은 TODO 앱 만들기. 이 시점에서 흥미를 느끼는 사람과 어려워하는 사람이 갈립니다. 어려워하는 사람을 위한 보조 자료를 미리 준비하세요.

5~6주: 네트워크 + API

Retrofit, JSON 파싱, Coroutine 기초. 핵심 팁: 실제 서비스의 API를 호출하는 실습이 가장 효과적입니다. 가짜 API보다 "우리 서비스의 진짜 데이터가 내가 만든 앱에 뜬다"는 경험이 몰입을 만듭니다.

7~8주: 아키텍처 + 실무 코드 리딩

MVVM, ViewModel, LiveData. 마지막 주는 실제 프로덕션 코드를 같이 읽으며 "왜 이 구조를 선택했는지" 설명합니다.

효과적인 교육을 위한 원칙

코드보다 "왜"가 먼저

RecyclerView를 가르칠 때 "ViewHolder 패턴은..."이 아니라 "스크롤할 때 매번 View를 새로 만들면 느려지니까"부터 시작하면 이해 속도가 다릅니다.

매주 동작하는 결과물

매주 끝날 때 "이번 주에 만든 것"이 있어야 합니다. 이론만 하면 3주차에 절반이 이탈합니다.

녹화보다 라이브

녹화 영상은 거의 안 봅니다. 라이브로 코딩하면서 실수하고 디버깅하는 과정을 보여주는 게 훨씬 효과적입니다.

핵심: 교육의 진짜 가치는 기술 전달이 아니라 공통 언어를 만드는 것입니다. "Activity", "Coroutine", "RecyclerView" 같은 단어를 팀 전체가 이해하면 커뮤니케이션 비용이 구조적으로 줄어듭니다.
