크래시율 100%에서 3%로 — SDK 관측 체계 구축기

모니터링이 전무한 SDK에 Sentry를 도입하고, 크래시 대응 프로세스를 체계화한 이야기.

현실: 모니터링 제로 상태

제가 합류했을 때 SDK의 크래시 모니터링은 존재하지 않았습니다. 크래시가 발생해도 알 방법이 없었고, 유일한 경로는 매체사에서 "SDK 때문에 앱이 죽는다"고 연락이 오는 것이었습니다.

왜 Sentry인가

Firebase Crashlytics도 검토했지만, SDK 안에 또 다른 Firebase SDK를 넣으면 매체사 앱과 의존성 충돌 위험이 있었습니다. Sentry는 독립적으로 동작하고, SDK 내부 초기화가 간단하며, 릴리스별 추적이 강력했습니다.

도입 과정

1. SDK 내부 Sentry 초기화

매체사 앱이 이미 Sentry를 사용할 수 있으므로, SDK 전용 Hub을 별도로 생성하여 충돌을 방지했습니다.

2. 릴리스 태깅

SDK 버전을 릴리스 태그로 전송. "이번 릴리스에서 크래시가 몇 건인지" 대시보드에서 바로 확인 가능.

3. 알림 체계

새 크래시 이슈 → Slack 즉시 알림. 임계값 초과 시 긴급 알림 별도 설정.

4. 대응 프로세스

  1. Triage: 영향 범위(사용자 수, 앱 수) 파악
  2. Priority: Critical / High / Medium 분류
  3. Fix: Critical은 핫픽스, High는 다음 정기 릴리스
  4. Postmortem: Critical 사후 분석 문서

대부분은 특정 Android 버전 WebView 초기화 실패, 네트워크 타임아웃 미처리, null 체크 누락이었습니다. 모니터링이 없었을 때는 이런 문제가 있는지조차 몰랐습니다.

핵심: SDK는 남의 앱 안에서 돌아갑니다. 내가 직접 사용자를 볼 수 없기 때문에 관측 체계가 곧 눈과 귀입니다.
