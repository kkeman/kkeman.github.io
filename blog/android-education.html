사내 Android 교육 8주 커리큘럼 설계기 — 13명의 개발자와 함께

비전공자부터 다른 플랫폼 개발자까지, 13명에게 8주간 Android 개발을 가르치며 배운 것.

배경

KT 나스미디어에서 SDK 개발을 하다 보면 다른 팀과 협업할 일이 많습니다. 그런데 SDK의 동작 원리를 이해하는 사람이 우리 팀 밖에는 거의 없었습니다. 백엔드 개발자가 "SDK에서 이 API를 왜 이렇게 호출하는지 모르겠다"고 하고, QA가 "이 크래시가 SDK 문제인지 앱 문제인지 구분을 못하겠다"고 하는 상황이 반복됐습니다.

교육의 목표는 "Android 개발자를 만드는 것"이 아니라, "SDK가 돌아가는 환경을 이해하는 사람을 늘리는 것"이었습니다.

수강생 구성

13명의 배경은 다양했습니다:

  • 백엔드 개발자 4명 (Java/Spring 경험)
  • 프론트엔드 개발자 2명 (React/Vue 경험)
  • QA 엔지니어 3명
  • 기획자 2명
  • 신입 개발자 2명

프로그래밍 경험이 전혀 없는 사람부터 다른 언어에 익숙한 사람까지 폭이 넓었습니다.

8주 커리큘럼

1~2주: Kotlin 기초 + Android 프로젝트 구조

변수, 함수, 클래스부터 시작. Android Studio 설치와 Hello World. 핵심은 "직접 빌드해서 폰에서 돌려보기"였습니다. 자기 폰에서 앱이 실행되는 경험이 동기를 만듭니다.

3~4주: UI 기초 + RecyclerView

Layout XML, View 시스템, RecyclerView로 리스트 표시. 실습은 간단한 TODO 앱 만들기. 여기서 절반이 "이거 재밌다"와 "이거 어렵다"로 갈립니다.

5~6주: 네트워크 통신 + API 연동

Retrofit으로 REST API 호출, JSON 파싱, 비동기 처리(Coroutine 기초). 실습은 우리 SDK의 실제 API를 호출해서 오퍼월 데이터를 가져와 표시하기. 이 시점에서 "아, SDK가 이런 식으로 데이터를 가져오는구나"를 체감합니다.

7~8주: 아키텍처 + SDK 구조 이해

MVVM 패턴, ViewModel, LiveData. 마지막 주는 실제 SDK 코드를 같이 읽으며 구조를 설명. 왜 Clean Architecture를 쓰는지, Dagger가 무슨 문제를 해결하는지.

효과가 있었나

직접적인 효과: QA팀이 크래시 리포트를 읽고 "이건 SDK의 RecyclerView Adapter 문제 같다"고 구체적으로 이야기할 수 있게 됐습니다. 백엔드 팀과 API 스펙을 논의할 때 "Android에서 이 형식은 파싱이 어렵다"는 피드백이 나오기 시작했습니다.

교육하면서 배운 것

1. 코드보다 "왜"가 먼저

비개발자에게 코드를 보여주기 전에 "이 코드가 해결하는 문제"를 먼저 설명해야 합니다. RecyclerView를 가르칠 때 "ViewHolder 패턴은..."이 아니라 "스크롤할 때 매번 View를 새로 만들면 느려지니까"부터 시작하면 이해 속도가 다릅니다.

2. 실습 환경 세팅이 반

첫 주에 Android Studio 설치하고 에뮬레이터 세팅하는 데만 2시간이 걸린 사람이 있었습니다. 사전에 환경 세팅 가이드를 배포하고, 첫 수업 전에 빌드가 되는 상태를 확인하는 게 중요합니다.

3. 20명 CI/CD 세미나도 진행

8주 교육 이후 CI/CD 세미나를 별도로 진행했습니다. GitHub Actions, 배포 자동화, Sentry 모니터링까지. 교육보다 세미나 형식이 경력 개발자에게는 더 효과적이었습니다. 이미 "왜"를 아는 사람들에게는 "어떻게"만 보여주면 됩니다.

조직에 남긴 것

교육은 일회성이지만, 공통 언어는 남습니다. "Activity", "Fragment", "Coroutine" 같은 단어를 팀 전체가 알고 있으면 커뮤니케이션 비용이 줄어듭니다. SDK 팀만의 용어가 아니라 조직의 공통 어휘가 되는 것, 그게 교육의 진짜 가치였습니다.
