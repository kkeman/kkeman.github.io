WebView JS Bridge 설계 — Native↔Web 통신 구조

SDK 안에서 Native와 Web이 통신해야 할 때, JS Bridge를 어떻게 설계하고 어떤 함정을 피해야 하는지.

왜 JS Bridge가 필요한가

오퍼월 SDK는 광고 목록을 WebView로 표시합니다. 광고 클릭, 포인트 적립 완료, 닫기 같은 이벤트는 Web에서 발생하지만, 처리는 Native에서 해야 합니다. 반대로 사용자 정보, 디바이스 ID 같은 데이터는 Native에서 Web으로 전달해야 합니다. 이 양방향 통신의 다리가 JS Bridge입니다.

설계 구조

Native → Web

Native에서 Web으로 데이터를 전달할 때는 evaluateJavascript를 사용합니다.

// Android (Kotlin)
webView.evaluateJavascript(
    "javascript:onNativeEvent('${eventJson}')",
    null
)

Web 쪽에서는 이 함수를 window에 등록해두면 됩니다.

// Web (JavaScript)
window.onNativeEvent = function(json) {
    const data = JSON.parse(json);
    handleEvent(data);
};

Web → Native

Web에서 Native로 이벤트를 보낼 때는 @JavascriptInterface를 사용합니다.

// Android (Kotlin)
class BridgeInterface {
    @JavascriptInterface
    fun onAdClick(adId: String) {
        // Native에서 광고 클릭 처리
    }

    @JavascriptInterface
    fun onClose() {
        // WebView 닫기
    }
}

webView.addJavascriptInterface(
    BridgeInterface(), "NativeSDK"
)
// Web (JavaScript)
window.NativeSDK.onAdClick("ad_12345");
window.NativeSDK.onClose();

실전에서 만난 함정들

1. 스레드 문제

@JavascriptInterface로 호출된 메서드는 WebView 스레드에서 실행됩니다. UI 업데이트나 Activity 조작을 하면 크래시가 발생합니다.

@JavascriptInterface
fun onClose() {
    // ❌ 크래시: WebView 스레드에서 UI 조작
    // activity.finish()

    // ✅ 메인 스레드로 전환
    handler.post { activity.finish() }
}

2. 타이밍 이슈

WebView 로딩이 완료되기 전에 evaluateJavascript를 호출하면 무시됩니다. onPageFinished 콜백 이후에 호출해야 합니다.

3. iOS와의 차이

iOS(WKWebView)에서는 WKScriptMessageHandler를 사용합니다. 인터페이스가 다르므로 Web 쪽 코드에서 플랫폼 분기가 필요합니다.

// Web에서 플랫폼 분기
function sendToNative(action, data) {
    if (window.NativeSDK) {
        // Android
        window.NativeSDK[action](JSON.stringify(data));
    } else if (window.webkit?.messageHandlers?.NativeSDK) {
        // iOS
        window.webkit.messageHandlers.NativeSDK
            .postMessage({ action, data });
    }
}

설계 원칙

Bridge 인터페이스는 SDK의 public API와 같습니다. 한 번 배포하면 매체사의 Web 페이지가 이 인터페이스에 의존하게 되므로 함부로 변경할 수 없습니다. 처음 설계할 때 버저닝을 고려하고, 메서드 시그니처를 신중하게 정해야 합니다.

배운 것

JS Bridge는 단순해 보이지만 디버깅이 어렵습니다. Native 쪽 로그와 Web 쪽 콘솔이 분리되어 있어서 통신 실패 시 어느 쪽 문제인지 파악하기 까다롭습니다. 저는 Bridge 호출마다 양쪽에 로그를 남기는 미들웨어 레이어를 추가해서 해결했습니다.
