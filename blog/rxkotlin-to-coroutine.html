RxKotlin에서 Coroutine으로 — 비동기 처리 마이그레이션 경험

Reactive Programming에서 Coroutine으로 전환한 과정. 왜 바꿨고, 어떤 점이 좋아졌고, 주의할 점은 무엇인지.

RxKotlin이란

Reactive Programming은 옵저버 패턴으로 데이터 스트림의 변경사항을 구독자에게 전달하는 프로그래밍 패러다임입니다. RxKotlin(RxJava의 Kotlin 확장)은 안드로이드에서 비동기 처리의 표준이었습니다.

왜 Coroutine으로 전환했나

RxKotlin은 강력하지만 학습 곡선이 높고 보일러플레이트가 많습니다. SDK 코드에서 특히 문제가 됐던 점들:

  • Disposable 관리 — CompositeDisposable을 빼먹으면 메모리 릭
  • 스레드 전환 — subscribeOn, observeOn 조합 실수로 인한 크래시
  • 디버깅 어려움 — 스택 트레이스가 RxJava 내부를 타고 다녀서 추적이 힘듦

Coroutine의 장점

// RxKotlin
api.getOffers()
    .subscribeOn(Schedulers.io())
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe({ offers ->
        updateUI(offers)
    }, { error ->
        handleError(error)
    })

// Coroutine
viewModelScope.launch {
    try {
        val offers = api.getOffers()
        updateUI(offers)
    } catch (e: Exception) {
        handleError(e)
    }
}

Coroutine은 순차적으로 읽히는 코드를 쓸 수 있게 합니다. try-catch로 에러를 잡고, structured concurrency로 생명주기를 자동 관리합니다.

마이그레이션 전략

  1. 새 코드는 Coroutine으로 — 기존 Rx 코드는 건드리지 않고 신규 기능만 Coroutine
  2. 점진적 전환 — 안정적인 영역부터 하나씩 교체
  3. Flow로 스트림 대체 — Observable/Flowable은 Flow로 전환
교훈: 마이그레이션은 한 번에 하지 않습니다. RxKotlin과 Coroutine은 공존할 수 있으므로 점진적으로 전환하는 것이 안전합니다. SDK에서는 특히 외부 API 호환성을 유지하면서 내부 구현만 바꾸는 전략이 중요합니다.
