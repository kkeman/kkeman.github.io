Kotlin Collection — Mutable vs Immutable, SDK에서의 선택 기준

List, Set, Map의 Mutable/Immutable 구분은 기본이지만, SDK 개발에서는 이 선택이 API 설계에 직결됩니다.

왜 SDK에서 Immutable이 중요한가

SDK의 public API에서 MutableList를 반환하면, 외부 개발자가 내부 상태를 직접 변경할 수 있습니다. 이건 버그가 아니라 설계 결함입니다.

// ❌ 외부에서 내부 상태 변경 가능
fun getOffers(): MutableList<Offer> = offers

// ✅ 읽기 전용 뷰만 노출
fun getOffers(): List<Offer> = offers.toList()

Collection 유형 정리

List — 순서 있는 컬렉션

// Immutable: 읽기만
val items = listOf("one", "two", "three")

// Mutable: 추가/삭제/변경
val items = mutableListOf(1, 2, 3)
items.add(4)
items.removeAt(0)

Set — 중복 없는 컬렉션

// 중복 자동 제거
val ids = setOf(1, 2, 3, 3, 2) // {1, 2, 3}

// Mutable
val ids = mutableSetOf<Int>()
ids.add(10)
ids.removeIf { it < 3 }

Map — 키-값 쌍

val config = mapOf("key1" to 1, "key2" to 2)

// Mutable
val config = mutableMapOf("one" to 1)
config["two"] = 2

SDK 설계 원칙

원칙: 내부는 Mutable로 관리하고, 외부에는 Immutable로 노출합니다. val에 MutableList를 할당해도 참조만 불변이지 내용은 변경 가능하므로, 외부 노출 시 반드시 toList()로 복사하거나 List 타입으로 캐스팅합니다.
