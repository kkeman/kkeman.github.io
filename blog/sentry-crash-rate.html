SDK에 Sentry를 도입할 때 알아야 할 것들

SDK 안에 크래시 모니터링을 넣을 때 도구 선택부터 매체사 앱과의 충돌 방지, 대응 프로세스 설계까지.

"문제가 없다"와 "문제를 모른다"는 다르다

SDK에 모니터링이 없으면 크래시가 발생해도 알 방법이 없습니다. 매체사(SDK를 탑재한 앱 운영사)에서 "앱이 죽는다"고 연락이 올 때까지 기다려야 합니다. 이건 문제를 뒤늦게 인지하는 것도 있지만, 문제의 규모를 파악할 수 없다는 게 더 치명적입니다.

도구 선택: 왜 Sentry인가

SDK 크래시 모니터링 도구를 선택할 때 세 가지를 고려해야 합니다:

  1. SDK 형태로 탑재 가능 — 우리 제품 자체가 SDK이므로 모니터링 도구도 SDK로 들어가야 합니다
  2. 이슈 그룹핑 — 같은 원인의 크래시를 묶고 우선순위를 매길 수 있어야 합니다
  3. 릴리스별 비교 — 새 버전 배포 후 크래시가 늘었는지 바로 확인 가능해야 합니다

Firebase Crashlytics도 검토했지만 SDK 안에 또 다른 Firebase SDK를 넣으면 매체사 앱과 의존성 충돌 위험이 있습니다. Sentry는 독립적으로 동작하고 릴리스 추적이 강력합니다.

SDK에 Sentry를 넣을 때 주의점

매체사 앱과의 Sentry 충돌

매체사가 이미 Sentry를 사용하고 있을 수 있습니다. SDK의 Sentry 인스턴스가 앱의 설정을 덮어쓰면 안 됩니다. 별도 Hub을 생성해서 격리해야 합니다.

// SDK 전용 Hub 생성 (앱의 Sentry와 격리)
val sdkHub = Hub(SentryOptions().apply {
    dsn = SDK_SENTRY_DSN
    release = SDK_VERSION
    environment = if (isDebug) "debug" else "production"
})

릴리스 태깅

SDK 버전을 릴리스 태그로 보내야 "이번 릴리스에서 크래시가 몇 건인지" 추적할 수 있습니다. CI/CD 파이프라인에서 버전을 자동으로 태깅하면 수동 실수를 방지할 수 있습니다.

알림 설계

모든 크래시에 알림을 보내면 금방 알림 피로가 옵니다. 두 단계로 나누는 게 효과적입니다:

  • 새 이슈 알림 — 처음 발생한 크래시 유형만 Slack 알림
  • 임계값 알림 — 특정 크래시가 N회 이상 발생하면 긴급 알림

대응 프로세스 설계

도구 도입은 30%이고, 프로세스 정의가 70%입니다. Sentry를 연동하는 건 하루면 되지만, 아래를 정해야 진짜 운영이 됩니다:

  1. Triage: 새 이슈 발생 시 영향 범위(사용자 수, 앱 수) 파악
  2. Priority: Critical(앱 강제 종료) / High(기능 불가) / Medium(부분 영향) 분류
  3. Fix: Critical은 핫픽스, High는 다음 정기 릴리스
  4. Postmortem: Critical 사후 분석 문서 작성

자주 발견되는 크래시 패턴

SDK에서 반복적으로 나오는 크래시 유형이 있습니다:

  • 특정 Android 버전 WebView 초기화 실패 — OS 파편화 문제
  • 네트워크 타임아웃 미처리 — try-catch 누락
  • null 체크 누락 — API 응답이 예상과 다를 때
  • 스레드 충돌 — WebView 스레드에서 UI 조작 시도
핵심: SDK는 남의 앱 안에서 돌아갑니다. 내가 직접 사용자를 볼 수 없기 때문에, 관측 체계가 곧 눈과 귀입니다. SDK 개발에서 모니터링은 선택이 아닙니다.
