배포 2시간을 20분으로 — GitHub Actions CI/CD 파이프라인 설계기

56개 앱에 SDK를 배포하는 파이프라인을 처음부터 구축한 과정. Maven Central, Firebase App Distribution, Confluence 릴리스 노트까지 자동화하며 배운 것들.

문제: 배포가 너무 오래 걸린다

KT 나스미디어에서 리워드 오퍼월 SDK를 개발하고 있습니다. 이 SDK는 하나머니를 포함한 56개 앱과 SSP에 탑재되어 있고, 일 사용자가 25만 명에 달합니다.

제가 합류했을 때 배포 프로세스는 전부 수동이었습니다. Android AAR을 Maven Central에 올리고, iOS를 CocoaPods에 push하고, TestApp을 Firebase로 배포하고, Confluence에 릴리스 노트를 쓰고, Slack으로 공지하는 것까지. 전체 소요 시간: 약 2시간. 단순 반복에 2시간을 쓰는 것도 문제지만, 사람이 하는 일이라 버전 태그를 빼먹거나 릴리스 노트에 이전 버전 내용을 복붙하는 실수가 실제로 있었습니다.

설계 원칙

  1. 한 번의 트리거로 끝까지 — git tag push 하나로 빌드부터 배포, 문서화까지 완료
  2. 실패하면 전체 중단 — Maven Central에는 올라갔는데 CocoaPods에는 안 올라간 상태 방지
  3. 사람 개입 최소화 — 승인이 필요한 단계만 수동으로

v1.2.3 형식의 git tag를 push하면 시작됩니다. 태그에서 버전을 파싱해서 이후 모든 단계에서 사용하므로 버전 불일치가 원천 차단됩니다.

on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'
삽질 포인트: Sonatype staging repository는 close 후 release까지 시간이 걸립니다. close 직후 release를 호출하면 실패합니다. polling으로 상태를 확인한 뒤 release하도록 수정했습니다.

배운 것

자동화의 핵심은 "사람이 실수할 수 있는 지점"을 제거하는 것입니다. 빠르게 하는 건 부수적 효과이고, 진짜 가치는 일관성에 있습니다. 같은 프로세스가 매번 동일하게 실행된다는 확신이 있으면 팀 전체의 배포 불안감이 사라집니다.

파이프라인은 처음부터 완벽할 필요가 없습니다. 저도 처음에는 Sonatype 배포만 자동화하고 나머지는 수동이었습니다. 점진적으로 단계를 추가하며 완성했습니다.
