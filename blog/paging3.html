JetPack Paging 3.0 — 오퍼월 SDK에서의 적용

Paging 3.0은 대량 데이터를 효율적으로 로드하는 라이브러리입니다. 오퍼월 SDK에서 수백 개의 광고 아이템을 로드할 때 어떻게 적용했는지 정리합니다.

왜 Paging이 필요한가

오퍼월 SDK는 한 화면에 수백 개의 광고 아이템을 표시해야 합니다. 전부 한 번에 로드하면 메모리 사용량이 급증하고 초기 로딩이 느려집니다. Paging 3.0은 필요한 만큼만 점진적으로 로드하여 이 문제를 해결합니다.

아키텍처

Paging 3.0은 세 개의 레이어로 구성됩니다:

  • Repository: PagingSource가 데이터 소스와 통신
  • ViewModel: Pager가 PagingData 스트림 생성
  • UI: PagingDataAdapter가 RecyclerView에 바인딩

핵심 코드

PagingSource 구현

class OfferPagingSource(
    private val api: OfferApi,
    private val query: String
) : PagingSource<Int, Offer>() {
    override suspend fun load(
        params: LoadParams<Int>
    ): LoadResult<Int, Offer> {
        return try {
            val page = params.key ?: 1
            val response = api.getOffers(query, page)
            LoadResult.Page(
                data = response.offers,
                prevKey = null,
                nextKey = response.nextPage
            )
        } catch (e: Exception) {
            LoadResult.Error(e)
        }
    }
}

ViewModel에서 Pager 생성

val offersFlow = Pager(
    config = PagingConfig(pageSize = 20)
) {
    OfferPagingSource(api, query)
}.flow.cachedIn(viewModelScope)

SDK에서의 고려사항

주의: SDK에서 Paging을 사용할 때는 매체사 앱의 Paging 버전과 충돌하지 않도록 의존성 관리에 주의해야 합니다. implementation 대신 compileOnly로 선언하고, 매체사가 직접 Paging 의존성을 추가하도록 가이드하는 것이 안전합니다.
