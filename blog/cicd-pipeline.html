SDK 배포 파이프라인을 설계할 때 고려할 것들

GitHub Actions로 Maven Central, Firebase App Distribution, Confluence까지 자동화하는 파이프라인을 구축하며 만난 기술적 판단들과 삽질.

수동 배포의 문제

SDK를 여러 패키지 레지스트리에 배포하는 과정이 전부 수동이면 두 가지 문제가 생깁니다. 첫째, 시간이 오래 걸립니다. 둘째, 사람이 하는 일이라 실수가 발생합니다. 버전 태그를 빼먹거나, 릴리스 노트에 이전 버전 내용을 복붙하는 일이 실제로 일어납니다.

자동화의 핵심은 속도보다 일관성입니다. 같은 프로세스가 매번 동일하게 실행된다는 보장이 있으면 배포에 대한 불안감이 사라집니다.

설계 시 핵심 결정들

트리거: Tag vs Branch vs Manual

세 가지 선택지가 있습니다. main 브랜치 push마다 배포하면 너무 자주 돌아가고, Manual dispatch는 결국 사람이 버튼을 눌러야 합니다. Git tag(v1.2.3)를 트리거로 쓰면 "배포 의도"가 명확하고, 태그에서 버전을 파싱할 수 있어서 버전 불일치가 원천 차단됩니다.

on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'

실패 전략: 전체 중단 vs 부분 계속

Maven Central에는 올라갔는데 CocoaPods에는 안 올라간 상태가 되면 곤란합니다. 저는 전체 중단 전략을 선택했습니다. 어떤 단계든 실패하면 이후 단계는 실행하지 않고 Slack으로 실패 알림을 보냅니다.

삽질 모음

Sonatype Staging Repository 타이밍

Sonatype에 publish할 때 staging repository를 close한 뒤 release를 해야 하는데, close 직후 바로 release를 호출하면 실패합니다. staging repository 상태가 "closed"로 전환되는 데 시간이 걸리기 때문입니다. polling으로 상태를 확인한 뒤 release하도록 수정해야 합니다.

GPG 서명 키 관리

Maven Central은 GPG 서명이 필수입니다. CI에서 GPG 키를 쓰려면 GitHub Secrets에 base64로 인코딩해서 저장하고, 파이프라인에서 디코딩해서 import해야 합니다. 키 만료일도 관리해야 하는데 이걸 놓치면 갑자기 배포가 실패합니다.

Confluence API 인증

Confluence REST API로 릴리스 노트를 자동 생성할 때, API 토큰 기반 인증을 사용합니다. 주의할 점은 Confluence Cloud와 Server의 API 엔드포인트가 다르다는 것입니다.

점진적으로 만들어라

처음부터 전체 파이프라인을 한 번에 만들려고 하지 마세요. 저도 처음에는 Sonatype 배포만 자동화하고 나머지는 수동이었습니다. 하나가 안정되면 다음 단계를 추가하는 식으로 점진적으로 완성했습니다. 중요한 건 완성도가 아니라 시작하는 것입니다.

핵심: 파이프라인은 "빠르게 배포하는 도구"가 아니라 "실수를 구조적으로 방지하는 시스템"입니다. 속도 향상은 부수적인 효과일 뿐입니다.
